# "X~Y 展开" 动作歧义陷阱 — 2026-06-29 事故复盘

## 事故

**用户原话**：`x8~9 展开，X12 通过`

**Agent 误读**：把 X8/X9 当成"通过"（与 X12 一起批了），用 `approve.mjs --force --skip-preaudit` 把 X8/X9 也批了出去。

**正确解析**：X8~9 = 展开 X8 到 X9 这两条详情；只有 X12 是通过。

**结果**：
- X12 沈煜 ¥18,867.92 投流费用 → **应批，已批** ✅
- X9 郑雨欣 ¥12,400 合同用印 → **误批**，薪福通后端审批链节点已置 AGREE，无法回退 ❌
- X8 汪培云 ¥1,678 员工日常报销 → `approve.mjs` 返 `ok=true clickVerified=true dbSaved=true` 但**薪福通后端实际未生效**（列表仍显示该单），可能是因为审批链后续节点的 cookie/权限检查失败

## 根因

1. **`~` 连接符歧义**：中文 `X8~9` 在"X号 + 动作"语境下不一定是范围通过，也可能是"范围 + 展开"
2. **省 token 思维**：Agent 推断"既然 X12 是通过，那 X8~9 大概率也是通过"，直接批量 `approve.mjs --force`
3. **SKILL 硬规则被忽略**：原 SKILL 已有"不得擅自扩大审批范围"（2026-05-02 事故），但执行时未触发拦截

## 硬规则（强制）

**任何包含「展开 / 详情 / 看下 / review / 拉一下」动词的指令，绝对不得调用 `approve.mjs`。**

### 复合指令解析流程

1. **Step 1**: 扫描指令中所有"通过 / 过 / 同意"关键词 + 关联 X 编号 → 归入"通过集合"
2. **Step 2**: 剩余 X 编号 + 动作 = 默认走"展开"，即使用户用了 `~` 或顿号连接
3. **Step 3**: 分别执行 — 展开的走 `review.mjs` / `navigate.mjs bill`，通过的走 `approve.mjs`
4. **Step 4**: 模糊场景（"X8~9 展开，X12 通过"无歧义，但"X1~3、X5 过"呢？→ X1/X2/X3 是展开还是通过？必须询问用户）

### 灰色地带处理

| 指令示例 | 解析 |
|---------|------|
| `X12 通过` | 单一动作，明确 |
| `x8~9 展开` | 范围 + 展开，明确 |
| `x8~9 展开，X12 通过` | 范围 + 展开 + 单一通过，明确 |
| `1、2、3 展开，4 通过` | 列表项动作并列，明确 |
| `X1~5` | **模糊**——是展开 5 条还是通过？必须问 |
| `上面那些都过` | "上面"指代不明——回查上一轮展示列表，明确每条 billId 后再执行 |
